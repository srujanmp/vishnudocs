📌 What to expect in DSA

OA focuses purely on Strings and Arrays. Questions are easy to easy-medium difficulty. Speed and correctness both matter — practice writing clean, bug-free code fast.

Key insight: Every array question can often be solved with sliding window, two-pointer, or prefix sums. Every string question usually involves hashing, two-pointer, or careful indexing. Know these cold.

CORE PATTERNS TO MASTER

Two Pointers
Left/right pointers converging. Handles sorted arrays, pair sums, palindrome checks, removing duplicates.
🔑 Valid Palindrome, 3Sum, Container With Most Water, Remove Duplicates
Sliding Window
Fixed or variable-size window over array/string. Optimal for subarray/substring problems.
🔑 Longest Substring Without Repeating, Max Sum Subarray of Size K, Minimum Window Substring
Prefix Sum
Precompute cumulative sums for O(1) range queries. Essential for subarray sum problems.
🔑 Subarray Sum Equals K, Product of Array Except Self, Range Sum Query
Frequency / Hashing
HashMap/HashSet for O(1) lookup. Group anagrams, find duplicates, intersection problems.
🔑 Group Anagrams, Two Sum, Longest Consecutive Sequence, Valid Anagram
Kadane's Algorithm
Maximum subarray problem. Classic DP on arrays reduced to O(n).
🔑 Maximum Subarray, Maximum Product Subarray, Circular Subarray
In-place Array Ops
Rotate, reverse, move elements — O(1) space. Crucial for OA where edge cases trip you up.
🔑 Rotate Array, Move Zeroes, Next Permutation, Sort Colors
String Matching & Parsing
KMP, Rabin-Karp for substring search. Also covers string builder, char manipulation.
🔑 Find All Anagrams in String, Repeated Substring Pattern, String to Integer
Stack-based String Ops
Monotonic stacks, bracket matching, expression evaluation using string iteration.
🔑 Valid Parentheses, Decode String, Largest Rectangle in Histogram (via string parsing)

📌 Round 2 DSA Focus Areas

Two major areas: Advanced String Techniques and Binary Search on Answers — a common but underrated pattern where the answer itself has a monotonic property you can binary search over.

Pro Tip: "Binary Search on Answers" problems are identified by the phrase "minimize the maximum" or "maximize the minimum" — as soon as you see this, think binary search on the answer space, not the array.
🔤 Advanced Strings Hard
  • KMP algorithm — failure function, O(n+m) search
  • Z-algorithm — string matching, period finding
  • Rabin-Karp — rolling hash for multiple pattern matching
  • Trie data structure — autocomplete, prefix search
  • Manacher's algorithm — all palindromic substrings in O(n)
  • Suffix Array — advanced, good to know conceptually
  • String hashing — polynomial hashing, collision handling
🔍 Binary Search on Answers Hard
  • Classic BS on sorted arrays — lo/hi/mid template
  • BS on answer: "Can we achieve X with K resources?"
  • Minimize maximum — split array largest sum
  • Maximize minimum — aggressive cows, EKO
  • Rotated sorted array — find pivot, search variant
  • BS on floating point — nth root, square root
  • Capacity to Ship Packages Within D Days

💡 Binary Search on Answers — Code Template
// Template: Binary Search on the Answer Space def solve(nums, k): lo, hi = min(nums), sum(nums) # answer range def feasible(mid): # Can we achieve "mid" as the answer? count, current = 1, 0 for num in nums: current += num if current > mid: count += 1 current = num return count <= k while lo < hi: mid = (lo + hi) // 2 if feasible(mid): hi = mid # try smaller else: lo = mid + 1 # need larger return lo

LOGICAL REASONING & RIDDLES — THEY CARE HOW YOU THINK

Critical insight: These questions are not about the answer — they're about your thinking process. Talk out loud. State assumptions. Narrow the solution space step by step. An elegant, methodical wrong answer beats a lucky correct one.
⚖️ Classic Weighing / Measurement Hard
  • Find odd ball in N with a balance scale in minimum weighings
  • Measure specific volume using oddly-sized containers
  • Weigh 9 balls, one heavier — minimum weighings needed?
  • Key technique: divide into thirds, not halves
💡 Light Bulb / Switch Puzzles Medium
  • 3 switches, 1 bulb in another room — find which switch
  • Turn on switch 1 for 5 mins, off, turn on switch 2, enter room
  • Key insight: Heat = third state beyond on/off
  • Generalizes to "use side effects as information"
🏃 Crossing the Bridge Problems Medium
  • 4 people, torch, bridge at night — minimize time
  • Always send 2 fastest together, return 1 fastest alone
  • Key: the bottleneck is moving the slowest people
  • Practice with 1,2,5,10 and 1,2,7,10 variants
📊 Probability & Expectation Hard
  • Expected number of coin flips to get HH vs HT
  • Gambler's ruin, secretary problem overview
  • Birthday paradox — understand intuitively
  • Monty Hall problem — Bayesian update logic
🧩 Logic / Truth Teller Puzzles Medium
  • Knights and Knaves — truth tellers and liars
  • "What would the other person say?" meta-question trick
  • 3-god problem (Random, True, False)
  • Blue eyes island logic puzzle — common knowledge
🎭 Fermi Estimation Easy
  • "How many golf balls fit in a school bus?" — structure your approach
  • Break into: volume ÷ ball volume × packing factor
  • Always start with a known anchor fact
  • Practice: piano tuners in NYC, fuel stations in India

    🏗️ System Design Framework — The 45-Min Template

    They asked you to design Instagram user page. Use this exact framework for any system design question.

    1️⃣Clarify Requirements (5 min)
    • Functional: What features exactly? (post feed, stories, followers, DMs?)
    • Non-functional: Scale? (100M users, 10M DAU) Latency? Consistency?
    • Out of scope: Clearly state what you're NOT designing
    • Read/write ratio — read-heavy (like social feeds) changes everything
    2️⃣Capacity Estimation (5 min)
    • DAU × requests/day = total QPS
    • Storage per user × total users = total storage
    • Read QPS vs Write QPS (social: ~100:1)
    • Bandwidth: image size × views/day
    3️⃣High-Level Design (15 min)
    • Client → CDN → Load Balancer → API Gateway
    • Microservices: User service, Feed service, Media service
    • Database choice: SQL for user data, NoSQL for feeds
    • Object store (S3) for images/videos
    • Message queue for async tasks (post processing, notifications)
    4️⃣Data Model & APIs (10 min)
    • Define key tables/schemas: users, posts, follows, likes
    • REST API endpoints for core features
    • Pagination strategy: cursor-based vs offset
    • Cache what you read most (user profiles, popular posts)
    5️⃣Deep Dive: Bottlenecks (10 min)
    • Feed generation: pull vs push vs hybrid model
    • Hot users (celebrities) — don't push to 10M followers on post
    • Media upload: presigned URLs, async processing pipeline
    • Global distribution: CDN, geo-replicated databases
    6️⃣Reliability & Scale
    • Database replication (master-slave) for read scaling
    • Sharding strategy: by user_id hash
    • Circuit breaker, rate limiting, bulkhead patterns
    • Monitoring: metrics, traces, alerting

    📸 Instagram User Page — Full Design Breakdown

    This is specifically what they asked you. Know this cold. Every component below is a potential follow-up question.

    • Profile data: Users table (user_id, username, bio, avatar_url, follower_count) — store in PostgreSQL, cache in Redis
    • Avatar images: Stored in S3/GCS, served via CDN (CloudFront). Thumbnail generation with Lambda on upload
    • Post grid: posts table (post_id, user_id, media_url, caption, created_at). Indexed by (user_id, created_at DESC)
    • Follower/Following counts: Maintain counters in Redis (fast reads). Async sync to DB. Problem: counter inconsistency during failures
    • Follow relationship: follows table (follower_id, followee_id, created_at). Index on both columns for bidirectional queries
    • Story highlights: highlight_reels table with ordered story references. Short TTL on actual story media
    • Privacy settings: Private accounts — every data read goes through a permission check. Cache permissions per (viewer, owner) pair
    • Infinite scroll for posts: Cursor-based pagination. cursor = (last_seen_post_id). Avoids offset degradation at scale
    • Optimistic UI: Like/follow updates immediately on client, async confirmed by server
    • CDN caching: Profile page = highly cacheable. Cache-Control: max-age=60. Invalidate on profile update

    🔬 Preparing for Project Deep Dive (45 min)

    For every project and internship on your resume, prepare the following structure. They WILL go deep — assume they know the domain.

    • Problem statement: What was the actual business/technical problem? Why did it matter?
    • Your specific contribution: Be precise — "I built X" not "we built X". Own your part fully.
    • Architecture decisions: Why this DB? Why this framework? What alternatives did you consider?
    • Technical challenges: Hardest bug, hardest design problem, what you had to learn on the fly
    • Scale & metrics: How many users? What was the QPS? Any performance improvements with numbers?
    • What you'd change: Every interviewer loves this. Shows maturity. Critique your own work.
    • What you learned: Technical AND soft skills. Shows growth mindset.
    Pro tip: Draw a diagram of your project architecture before the interview. Being able to sketch it instantly impresses interviewers and shows deep ownership.
